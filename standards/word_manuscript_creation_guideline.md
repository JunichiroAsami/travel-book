## 2. タイトル決定プロセス

### 2.1. 概要

書籍タイトルは、SEO対策、実用性、差別化が明確で、ターゲット層への訴求力が高いものを選定する必要があります。複数のAIの意見を取り入れることで、客観的な評価が得られます。

### 2.2. 標準プロセス

```
1. GeminiとChatGPTに複数のタイトル案を依頼（各10個）
   ↓
2. タイトル案の比較分析レポートを作成
   ↓
3. ユーザーが気に入ったタイトル案を選定
   ↓
4. 選定したタイトル案について、ChatGPTに評価を依頼
   ↓
5. ChatGPTの推奨する改善案について、Geminiにも評価を依頼
   ↓
6. 両AIから高評価を獲得したタイトルを最終決定
```

### 2.3. 工数見積もり

| ステップ | 工数 |
|:---|:---|
| 1. タイトル案の収集 | 0.5時間 |
| 2. タイトル案の比較分析 | 0.5時間 |
| 3. ユーザーによるタイトル選定 | 0.2時間 |
| 4. ChatGPTによるタイトル評価 | 0.3時間 |
| 5. 改善案の作成 | 0.3時間 |
| 6. Geminiによるタイトル評価 | 0.3時間 |
| 7. 最終タイトル決定 | 0.2時間 |
| **合計** | **2.3時間** |

### 2.4. 成果物

- `title_comparison_report.md` - タイトル案の比較分析レポート
- `gemini_title_proposals.md` - Geminiからのタイトル案
- `chatgpt_title_proposals.md` - ChatGPTからのタイトル案
- `title_final_decision_report.md` - 最終タイトル決定レポート
- `title_comparison_gemini_chatgpt.md` - GeminiとChatGPTの評価比較

### 2.5. 評価基準

タイトルは以下の観点で評価します：

1. **SEO対策**: 検索されやすいキーワードが含まれているか
2. **実用性**: 本の内容が一目でわかるか
3. **差別化**: 他の書籍との違いが明確か
4. **ターゲット層への訴求力**: 読者に響くか
5. **キャッチー度**: 目を引くか

## 3. 表紙・奥付・前付け後付けの整備

### 3.1. 概要

書籍として完成させるには、表紙、奥付、前付け（前書き、中扉）、後付け（後書き、ディスクレーマー）が必要です。これらの要素は個別に管理し、最終的にWord原稿に統合します。

```

   ↓

4. 参考資料のフォーマット分析

5. ディスクレーマーと奥付の再作成（参考資料の形式に準拠）
   ↓
6. 前書き・中扉・後書きの作成（Geminiに依頼）
   ↓
7. 各要素をMarkdownファイルとして個別管理
   ↓

```

### 3.5. 書籍の構成順序

正しい構成順序は以下の通りです：

1. **扉**（表紙の次のページ）
2. **前書き**（まえがき）
3. **目次**
4. **中扉**（本文の開始を示すページ）
5. **本文**（第1章〜第N章）
6. **後書き**（あとがき）
7. **索引**（オプション）
8. **ディスクレーマー**（免責事項）
9. **奥付**

**注意**: この順序を間違えると、読者が混乱します。必ず確認してください。

## 4. Word原稿の作成

### 4.1. 概要

Word原稿の作成は、各要素を個別にWord化してからマージする方法が最も確実です。統合ファイルから部分的にコピーする方法は、エラーが発生しやすいため避けてください。

### 4.2. 標準プロセス

```
1. 各要素をMarkdownファイルとして個別管理
   ↓
2. 各要素をWord形式に変換
   ↓
3. 正しい順番でマージ
   ↓
4. PDF化して構成順序を確認
   ↓
5. 問題があれば修正
```

### 4.3. 工数見積もり

| ステップ | 工数 |
|:---|:---|
| 1. 各要素をMarkdownファイルとして個別管理 | 0.5時間 |
| 2. 各要素をWord形式に変換 | 1時間 |
| 3. 正しい順番でマージ | 1時間 |
| 4. PDF化して構成順序を確認 | 0.5時間 |
| 5. 問題があれば修正 | 1時間 |
| **合計** | **4時間** |

### 4.4. 成果物

- `complete_manuscript_vXX.docx` - Word原稿
- `complete_manuscript_vXX.pdf` - PDF原稿（確認用）

### 4.5. 注意事項

#### 4.5.1. 段落のコピーは範囲指定では行わない

**問題**:
```python
# ❌ 悪い例：範囲指定で段落をコピー
for para in doc.paragraphs[155:]:
    new_doc.add_paragraph(para.text)
```

この方法では、段落の大部分が失われる可能性があります。

**解決策**:
```python
# ✅ 良い例：各要素を個別にWord化してからマージ
# 1. 扉をWord化
title_page_doc = create_title_page()
# 2. 前書きをWord化
preface_doc = markdown_to_word("preface_v1.md")
# 3. 目次をWord化（既存のv27から抽出）
toc_doc = extract_toc_from_v27()
# 4. 中扉をWord化
halftitle_doc = markdown_to_word("halftitle_v1.md")
# 5. 本文をWord化（既存のv27から抽出）
body_doc = extract_body_from_v27()
# 6. 後書きをWord化
afterword_doc = markdown_to_word("afterword_v1.md")
# 7. ディスクレーマーをWord化
disclaimer_doc = markdown_to_word("disclaimer_v2.md")
# 8. 奥付をWord化
colophon_doc = markdown_to_word("colophon_v2.md")

# 9. 正しい順番でマージ
final_doc = merge_documents([
    title_page_doc,
    preface_doc,
    toc_doc,
    halftitle_doc,
    body_doc,
    afterword_doc,
    disclaimer_doc,
    colophon_doc
])
```

#### 4.5.2. 作成後は必ずPDF化して確認

Word原稿を作成した後は、必ずPDF化して構成順序を確認してください。

**確認項目**:
1. 扉が最初に来ているか
2. 前書きが扉の後に来ているか
3. 目次が前書きの後に来ているか
4. 中扉が目次の後に来ているか
5. 本文が中扉の後に来ているか
6. 後書きが本文の後に来ているか
7. ディスクレーマーが後書きの後に来ているか
8. 奥付が最後に来ているか

**PDF化の方法**:
```bash
libreoffice --headless --convert-to pdf complete_manuscript_vXX.docx
```

## 5. 発生した課題と解決策

### 5.1. 課題1: Word原稿の作成時に段落が失われる

**問題の詳細**:
- v30作成時、v29から本文をコピーする際に、段落インデックスの指定が不正確だった
- 結果として、2447段落→102段落に減少し、本文の大部分が失われた

**原因**:
- `doc.paragraphs[155:]`のような範囲指定では、段落のコピーが正しく行われない
- python-docxライブラリの制約により、段落の範囲コピーが困難

**解決策**:
- 各要素を個別にWord化し、正しい順番でマージする方法に変更
- v27から目次と本文を抽出し、新しい要素（扉、前書き、中扉、後書き、ディスクレーマー、奥付）と統合

**教訓**:
- **Word原稿の作成は、各要素を個別にWord化してからマージする方法が最も確実**
- 統合ファイルから部分的にコピーする方法は、エラーが発生しやすい

### 5.2. 課題2: 構成順序の間違い

**問題の詳細**:
- v28作成時、奥付が最初に来ていた
- v29作成時、中扉が本文の後に配置されていた

**原因**:
- 各要素の追加順序を間違えた
- python-docxライブラリでは、要素の挿入位置を明示的に指定する必要がある

**解決策**:
- 正しい構成順序を明確に定義し、その順序通りに要素を追加
- 作成後、必ずPDF化して構成順序を確認

**教訓**:
- **Word原稿の作成後は、必ずPDF化して構成順序を確認する**
- 構成順序のチェックリストを作成し、毎回確認する

### 5.3. 課題3: ビジュアル面が全くできていない

**問題の詳細**:
- プロンプトと回答の区別が不明瞭（視覚的な区別が一切ない）
- 画像が1枚しかない（扉の画像のみ）
- 表が1つもない
- スタイルの使用が単調（99.5%が「Normal」スタイル）
- 太字・斜体の使用が不十分（斜体が0箇所）

**原因**:
- MarkdownからWordへの変換時に、視覚的な要素が失われた
- プロンプトと回答の区別を、Markdownの引用記法（`>`）で表現していたが、Wordでは単なるインデントになった

**解決策**:
- nano banana（Gemini画像生成）で、チャット画面風のUI画像を生成
- 各章のMDファイルに、画像と表を追加
- プロンプトと回答を視覚的に区別するスタイルを作成（背景色、枠線など）

**教訓**:
- **MarkdownからWordへの変換では、視覚的な要素が失われる**
- **プロンプトと回答の区別は、Wordのスタイル機能で実装する必要がある**
- **各章のMDファイルに、画像と表を追加してから、Wordに変換する**

## 6. ビジュアル面の改善

### 6.1. 概要

ビジュアル面の改善は、読者の理解を助け、読みやすさを向上させるために重要です。特に、プロンプトと回答の視覚的区別は、AI活用書では必須です。

### 6.2. 優先度

| 優先度 | 項目 | 工数 |
|:---|:---|:---|
| 優先度1 | プロンプトと回答の視覚的区別 | 2〜3時間 |
| 優先度2 | 画像の追加 | 1〜2時間 |
| 優先度3 | 表の追加 | 1〜2時間 |
| 優先度4 | スタイルの統一 | 1時間 |
| 優先度5 | 太字・斜体の適用 | 30分 |

### 6.3. 実装方法

#### 6.3.1. プロンプトと回答の視覚的区別

**方法1: Wordのスタイル機能を使用**

1. プロンプト用のスタイルを作成
   - 背景色: 薄い青（RGB: 230, 240, 255）
   - 枠線: 実線、青色（RGB: 0, 112, 192）
   - フォント: メイリオ、10pt

2. 回答用のスタイルを作成
   - 背景色: 薄い緑（RGB: 230, 255, 230）
   - 枠線: 実線、緑色（RGB: 0, 176, 80）
   - フォント: メイリオ、10pt

**方法2: nano banana（Gemini画像生成）を使用**

1. チャット画面風のUI画像を生成
2. プロンプトと回答を吹き出しで表現
3. 画像を各章のMDファイルに挿入

#### 6.3.2. 画像の追加

1. 旅行先の写真を各章に挿入
2. AIの回答画面のスクリーンショットを挿入
3. 図表やイラストを挿入（nano bananaで生成）

#### 6.3.3. 表の追加

1. AIツール比較表を作成
2. 旅程表を作成
3. 費用の内訳表を作成
